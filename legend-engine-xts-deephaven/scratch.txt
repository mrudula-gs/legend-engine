Summary
Legend is a premier data modeling and execution platform optimized for static datasets, delivering finite results such as tabular datasets from single snapshots. However, the increasing prevalence of streaming data—continuously updating datasets from systems like Deephaven (ticking tables) and Chunkstore (streaming feeds)—requires capabilities beyond Legend’s current batch-processing scope. This limitation restricts Legend from supporting real-time analytics and dynamic pipelines, critical for modern data-driven applications.

This proposal presents a framework to extend Legend with streaming data support, leveraging its existing stereotype system and integrating store-native APIs for optimal performance. Key objectives include:

Modeling streaming data in Pure with appropriate metadata.
Enhancing Pure queries with streaming and timeseries operators.
Executing queries using store-native streaming mechanisms.
Delivering and visualizing continuous results in Legend Studio.
Using Deephaven as a primary example, the design ensures flexibility for Chunkstore and other stores, enhancing Legend’s versatility while preserving its core architecture.

Introduction
Background
Legend’s Pure language and execution framework excel at modeling static datasets, utilizing stereotypes (e.g., <<meta::pure::profiles::temporal.businesstemporal>>) to extend metadata. However, streaming data from sources like Deephaven’s nds_desktops or Chunkstore’s timeseries feeds requires native support for continuous updates, which Legend currently lacks.

Motivation
Streaming support enables:

Real-Time Insights: Live monitoring of desktop statuses via Deephaven’s nds_desktops.
Dynamic Pipelines: Continuous timeseries analysis from Chunkstore feeds.
Unified Platform: Seamless integration of static and streaming workflows in Legend.
Without this capability, Legend remains constrained to static use cases, trailing platforms optimized for real-time data processing, such as Deephaven or Kafka Streams.

Objectives
Model streaming data using Legend’s stereotype system with class-level metadata.
Extend Pure queries for streaming and timeseries operations.
Integrate store-native streaming APIs into the execution framework.
Deliver continuous results with enhanced visualization in Legend Studio.
Proposed Framework
1. Modeling Streaming Data
Objective: Enable Pure to model streaming data using existing stereotypes and tagged values at the class level.

Approach: Define a streaming stereotype in a custom profile, applying tagged values like timeseriesKey to class properties and storeType to classes, as tags cannot be applied to stores. Use the provided Deephaven store definition as the streaming source.

Details:

Profile Definition:
pure

Wrap

Copy
Profile meta::pure::profiles::streaming
{
  stereotypes: [streaming];
  tags: [timeseriesKey, storeType];
}
Stereotypes: <<meta::pure::profiles::streaming.streaming>> marks a class or store as streaming, indicating continuous updates.
Tagged Values:
{meta::pure::profiles::streaming.timeseriesKey: 'propertyName'}: Applied to a property (e.g., eventTime), identifies it as the temporal key for timeseries operations, providing explicit context rather than assuming all DateTime fields are relevant.
{meta::pure::profiles::streaming.storeType: 'Deephaven'}: Applied at the class level, specifies the intended store type, as stores themselves cannot hold tags.
Store Definition: Use the existing Deephaven store syntax without tagged values, relying on class-level metadata.
TimeseriesKey: Designates the property used for temporal ordering or windowing in timeseries queries, ensuring clarity and intent.

2. Mapping Streaming Data
Objective: Map streaming classes to store sources using existing mechanisms.

Approach: Utilize Legend’s standard mapping syntax, with store-specific details inferred from class-level storeType tags.

Details:

Existing Syntax: No new mapping constructs are needed; standard mappings suffice.
Studio Support: UI recognizes storeType from the class (e.g., "Deephaven") to streamline mapping.

3. Connection and Runtime
Objective: Configure streaming store access and runtime execution.

Approach: Define StreamingStoreConnection for basic connectivity, with subscription options (e.g., refresh rate) specified at the query level for flexibility.

Details:

Connection: Provides store type, URL, and authentication, kept generic as connections support multiple queries.
Query-Level Options: tick in queries (e.g., tick('1s')) defines refresh rates, aligning with user intent per query rather than fixing it at the connection level.
StreamingRuntime: Links mappings to connections.

4. Query Language and Platform API
Objective: Extend Pure for streaming and timeseries queries with a clear API.

Approach: Add tick for streaming updates and window for timeseries aggregation, integrating with Legend’s groupBy and agg, with subscribe as the API for continuous execution.

Details:

Tick: Specifies update frequency (e.g., tick('1s')), streaming results continuously.
Window: Aggregates over time intervals (e.g., window('1 minute', $this.eventTime)), paired with groupBy and agg for timeseries analysis.
API (subscribe): Returns a Stream for real-time consumption, distinct from static execute.

Function model::PersonQuery(): Stream<meta::pure::tds::TabularDataSet>[1]
{
  model::Person.all()
    ->filter(x | $x.employeeType == model::EmployeeType.FULL_TIME)
    ->project([x | $x.firstName, x | $x.lastName], ['First Name', 'Last Name'])
    ->tick('1s')
    ->from(execution::StreamingMapping, execution::StreamingRuntime)
}
Use Case: Streams full-time employees’ names every second for a live dashboard.

Function model::PersonWindowed(): Stream<meta::pure::tds::TabularDataSet>[1]
{
  model::Person.all()
    ->window('1 minute', $this.eventTime)
    ->groupBy(
      [x | $x.firstName],
      meta::pure::functions::collection::agg('count', x | $x.id, y | $y->count()),
      ['First Name', 'Count']
    )
    ->tick('10s')
    ->from(execution::StreamingMapping, execution::StreamingRuntime)
}
Use Case: Counts employees by first name in 1-minute windows, updating every 10 seconds, for trend analysis.

5. Execution Framework
Objective: Execute streaming queries using store-native APIs.

Approach: Extend Legend’s execution framework to integrate Deephaven’s streaming API conceptually.

Details:

Deephaven API: Uses BarrageSession.subscribe(TableSpec) to subscribe to ticking table updates, returning a FlightStream of continuous data. Legend translates Pure queries into a TableSpec, which Deephaven streams in real-time.
Integration Concept:
Query to Node: Pure queries (e.g., tick('1s')) become StreamingExecutionNode instances.
Executor: Invokes subscribe, passing the TableSpec, and wraps the FlightStream in a Stream.
Legend Fit: Extends existing node execution, adding a streaming path.
Use Case: Streams nds_desktops updates every second to Studio via Deephaven’s native API.

6. Result Handling
Objective: Deliver and visualize continuous streaming results.

Approach: Implement ContinuousStreamingResult with a store-provided cancellation mechanism and enhance Studio for real-time display.

Details:

Result Handling: Returns a Stream with a cancelSubscription action (e.g., BarrageSession.unsubscribe()), triggered by Studio UI controls, not modeled by users.
Studio Visualization:
Proposed: Real-time grid widget updates rows per tick, with "Pause" (buffer data), "Resume", and "Stop" (cancels subscription) buttons.
Use Case: Displays PersonQuery results updating every second, with user control over the stream.
Benefits
Aligned Design: Uses Legend’s stereotypes and mappings.
Flexible Queries: Query-level tick and window options.
Efficient Execution: Leverages store-native APIs.
Enhanced UI: Real-time visualization in Studio.
