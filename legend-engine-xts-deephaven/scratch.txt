Deephaven’s ticking tables are inherently dynamic, updating in real-time as new data arrives (e.g., trades in tradeEvents). Based on Deephaven’s documentation and typical behavior:

Update Frequency (e.g., refreshRate): Ticking tables update as data changes on the server, driven by the underlying data source (e.g., market feeds). There’s no fixed "update frequency" baked into the table itself—updates occur as events happen. Clients (like Legend) subscribing via BarrageSession.subscribe(TableSpec) receive these updates as they occur, with no explicit server-side polling interval. However, the client can control how often it processes or fetches updates (e.g., throttling or batching), which aligns with our tick('1s') query operator rather than a store-level setting.
Windowing Rules: Windowing (e.g., aggregating over 5 minutes) isn’t inherent to the table’s metadata—it’s a query operation applied to the stream. Deephaven supports windowed operations (e.g., via its query language), but these are defined at query time, not as table metadata.
Batch Size (e.g., batchSize): The number of rows fetched per update isn’t typically a table property in Deephaven—it’s a client-side subscription parameter, configurable via the BarrageSession API (e.g., buffering or batching options). This could vary by query or client needs, not fixed in the table definition.
Start Time (e.g., startTime): Ticking tables are live by default, starting from the current moment unless a historical replay or snapshot is requested. Deephaven allows specifying a start time for historical data via query filters (e.g., where("tradeTime >= '2025-01-01'")), but this isn’t metadata embedded in the table—it’s a query constraint.
