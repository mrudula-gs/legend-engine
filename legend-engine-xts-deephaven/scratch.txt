Legend is a leading data modeling and execution platform optimized for static datasets, delivering finite results such as tabular datasets from single snapshots. However, the rise of streaming data—continuously updating datasets from systems like Deephaven (ticking tables) and Chunkstore (streaming feeds)—demands capabilities beyond Legend’s current batch-processing focus. This limitation restricts Legend from supporting real-time analytics and dynamic pipelines, critical for modern data applications.

This proposal outlines a framework to extend Legend with streaming data support, leveraging its existing stereotype system and integrating store-native APIs. Key objectives include:

Modeling streaming data with store-level metadata in Pure.
Enhancing Pure queries with streaming and timeseries operators.
Executing queries using store-native streaming mechanisms.
Delivering and visualizing continuous results in Legend Studio.
Using Deephaven as a primary example with a tradeEvents streaming table, the design ensures flexibility for Chunkstore and other stores, maintaining Legend’s store-agnostic modeling principles.

Introduction
Background
Legend’s Pure language excels at modeling static datasets, using stereotypes (e.g., <<meta::pure::profiles::temporal.businesstemporal>>) to extend metadata. However, streaming data—such as Deephaven’s real-time trade events or Chunkstore’s timeseries feeds—requires native support for continuous updates, which Legend currently lacks.

Motivation
Streaming support enables:

Real-Time Insights: Live tracking of financial trades via Deephaven’s tradeEvents.
Dynamic Pipelines: Continuous analysis of timeseries feeds from Chunkstore.
Unified Platform: Seamless static and streaming workflows in Legend.
Without this, Legend trails platforms like Deephaven or Kafka Streams in real-time data processing.

Objectives
Model streaming data with metadata at the store level using stereotypes.
Extend Pure queries for streaming and timeseries operations.
Integrate store-native streaming APIs into the execution framework.
Deliver continuous results with Studio visualization support.
Proposed Framework
1. Modeling Streaming Data
Objective: Enable Pure to model streaming data with metadata confined to the store level.

Approach: Define a streaming stereotype in a custom profile, applying it to Store definitions with tagged values like storeType. Classes remain store-agnostic, with timeseriesKey tags on properties for temporal context.

Details:

Profile Definition:

Profile meta::pure::profiles::streaming
{
  stereotypes: [streaming];
  tags: [timeseriesKey, storeType];
}
Stereotypes: <<meta::pure::profiles::streaming.streaming>> marks a Store as streaming, indicating it provides continuous updates.
Tagged Values:
{meta::pure::profiles::streaming.storeType: 'Deephaven'}: Applied to the Store, specifies the store type.
{meta::pure::profiles::streaming.timeseriesKey}: Applied to a property (e.g., tradeTime), flags it as the temporal key for timeseries operations, avoiding redundant naming by using a standalone tag.
Store-Agnostic Classes: Classes do not include store-specific metadata, ensuring flexibility across stores.

4. Query Language and Platform API
Objective: Extend Pure for streaming and timeseries queries with a clear API.

Approach: Add tick for streaming and window for timeseries, integrating with groupBy and agg, with subscribe as the streaming API.

Details:

Tick: Streams updates at a specified interval (e.g., tick('1s')).
Window: Aggregates over time intervals using the timeseriesKey property.
API (subscribe): Returns a continuous Stream, enabling real-time processing.
Query Examples:

Function model::TradeQuery(): Stream<meta::pure::tds::TabularDataSet>[1]
{
  model::Trade.all()
    ->filter(x | $x.quantity > 100)
    ->project([x | $x.symbol, x | $x.price], ['Symbol', 'Price'])
    ->tick('1s')
    ->from(execution::StreamingMapping, execution::StreamingRuntime)
}
Use Case: Streams large trades every second for real-time market monitoring.

Function model::TradeVolume(): Stream<meta::pure::tds::TabularDataSet>[1]
{
  model::Trade.all()
    ->window('5 minutes', $this.tradeTime)
    ->groupBy(
      [x | $x.symbol],
      meta::pure::functions::collection::agg('totalQty', x | $x.quantity, y | $y->sum()),
      ['Symbol', 'Total Quantity']
    )
    ->tick('30s')
    ->from(execution::StreamingMapping, execution::StreamingRuntime)
}
Use Case: Aggregates trade volumes by symbol over 5-minute windows, updating every 30 seconds, for market trend analysis.

5. Execution Framework
Objective: Execute streaming queries using store-native APIs.

Approach: Extend Legend’s execution framework to integrate Deephaven’s streaming API conceptually.

Details:

Deephaven API: BarrageSession.subscribe(TableSpec) subscribes to ticking updates, returning a FlightStream of real-time data. Legend translates Pure queries into a TableSpec for subscription.
Integration Concept:
Pure query → StreamingExecutionNode → subscribe invocation → FlightStream → Stream.
Reuses Legend’s execution pipeline, adding a streaming branch.
Use Case: Streams tradeEvents updates every second to Studio via Deephaven’s API.

6. Result Handling
Objective: Deliver and visualize continuous streaming results.

Approach: Use a streaming result object with store-provided cancellation, enhancing Studio for real-time display.

Details:

Result Handling: Delivers a Stream with a cancelSubscription action (e.g., BarrageSession.unsubscribe()), triggered by Studio UI.
Studio Visualization: Real-time grid widget updates rows per tick, with "Pause", "Resume", and "Stop" controls.
Use Case: Displays TradeQuery results updating every second, with user control over the stream.

Benefits
Store-Agnostic Models: Classes remain flexible across stores.
Query Flexibility: tick and window at query level.
Efficient Execution: Leverages store-native APIs.
Enhanced UI: Real-time visualization in Studio.
